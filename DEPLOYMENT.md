# Cloud District Club — Deployment Guide

## Production-Safe Deployment Rules

### Rule 1 — Only source code is committed to GitHub

`frontend/dist` is **never committed** to the repository.
The production build is generated automatically during every deployment.

Build command:
```bash
bash scripts/build.sh
# or manually:
cd frontend && expo export --platform web
```

FastAPI serves the compiled output from:
```
/frontend/dist
```

### Rule 2 — All deployments originate from `main` only

- No `conflict_*` branches should ever be used for production deployments.
- Before deploying, confirm you are on `main`:
  ```bash
  git branch --show-current   # must output: main
  git log --oneline -3        # confirm the expected commits are at HEAD
  ```

### Rule 3 — Every stable deployment receives a git tag

Tag format:
```
v<major>.<minor>-<feature-slug>
```

Examples:
```
v1.0-stable-admin
v1.1-loyalty-system
v1.2-checkout-launch
```

Create a tag:
```bash
git tag v1.1-loyalty-system
# Then push via: Save to Github (Emergent platform)
```

## Build Version Footer

The admin panel footer displays the current build version:

```
Build YYYY-MM-DD-<commit-count>
```

This value is automatically generated by `scripts/build.sh` and baked into
the Expo bundle as `EXPO_PUBLIC_BUILD_VERSION` at export time.

In development/preview, the footer shows the fallback value: `2026-03-05-2`.

---

## Deployment Checklist

Before every production deployment:

- [ ] All changes committed and pushed to `main`
- [ ] No `conflict_*` branches referenced
- [ ] `frontend/dist` is NOT in the git commit (check `.gitignore`)
- [ ] Backend health check passes: `GET /health → 200 {"status":"ok"}`
- [ ] Run `bash scripts/build.sh` to regenerate `dist/`
- [ ] Create a git tag for the release

---

## API Configuration

All environments (preview, development, production) use the same backend:

```
EXPO_PUBLIC_BACKEND_URL=https://vape-shop-staging.emergent.host
```

This ensures:
- Order status updates appear in all environments
- Product images uploaded in production appear everywhere
- Cloudz balances, user accounts, and inventory are synchronized

## File Structure

```
/app
├── backend/          # FastAPI — serves /health, /api/*, and /frontend/dist
├── frontend/         # Expo (React Native Web)
│   ├── app/          # Screen source files
│   ├── components/   # Shared components
│   ├── dist/         # ← Generated during deployment. NEVER committed.
│   └── .gitignore    # Includes dist/
└── scripts/
    └── build.sh      # Production build script
```
